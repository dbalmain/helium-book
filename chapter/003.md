# Chapter 3: Functions

By the end of this chapter, with remarkably few changes, we'll be able to evaluate expressions like `let double = \x -> x + 2 in double 5`. That means lambdas, closures, and function application – the core of the lambda calculus. Helium is still dynamically typed at this point; we'll add a type checker in the next chapter.

> [!NOTE]
> The complete code for this chapter is on the [`chapter/003`](https://github.com/dbalmain/helium-lang/tree/chapter/003) branch.

Adding functions requires three structural changes:

1. Two new **AST nodes** – `Lam` for lambdas and `App` for application.
2. **Application parsing** – `f x` is written as juxtaposition, like in Haskell (no parens required), which needs careful handling in the parser.
3. A new **value type** and **closures** – `Int` is no longer enough, and functions need to capture the environment where they were defined.

## New AST nodes

Two new constructors in the syntax tree:

```haskell
-- src/Helium/Syntax.hs
data Expr
  = Lit Int
  | Var String
  | Let String Expr Expr
  | Lam String Expr       -- \x -> body
  | App Expr Expr         -- f x
  | Neg Expr
  | Add Expr Expr
  | Sub Expr Expr
  | Mul Expr Expr
  | Div Expr Expr
  deriving (Show, Eq)
```

`Lam "x" body` represents `\x -> body` – a lambda that takes a parameter named `x` and evaluates `body` when called.

> [!NOTE]
> As with `let`, we're using Haskell syntax here. I want to change this later, but it's helpful to use the familiar at this stage.

`App f x` represents the application of function `f` to argument `x`. The expression `double 5` becomes `App (Var "double") (Lit 5)`. Multi-argument application like `add 3 4` is handled by nesting: `App (App (Var "add") (Lit 3)) (Lit 4)`.

## Parsing lambdas

Lambda syntax is `\x -> body`. The parser for it is similar in shape to the `letExpr` parser:

```haskell
lambdaExpr :: Parser Expr
lambdaExpr = do
  _ <- symbol "\\"
  parameter <- identifier <?> "parameter"
  _ <- symbol "->"
  Lam parameter <$> expr
```

Lambda lives at the `expr` level, not inside `term` or `atom`. This is intentional. It means the body extends as far right as possible. `\x -> x + 1` is parsed as `Lam "x" (Add (Var "x") (Lit 1))`, not as `App (Lam "x" (Var "x")) (Lit 1)`. To pass a lambda as an argument, you need parens: `f (\x -> x)`.

The updated `expr` tries each alternative in order:

```haskell
expr :: Parser Expr
expr = letExpr <|> lambdaExpr <|> makeExprParser term operatorTable
```

## Parsing application

In Helium, function application is written by juxtaposition: `f x`, not `f(x)`. This is standard in ML-family languages. The trick is handling it without a dedicated operator token.

We achieve this by splitting what used to be called `term` into two levels.

An **atom** is a single irreducible unit – a parenthesised expression, an integer literal, or a variable reference:

```haskell
atom :: Parser Expr
atom = parens expr <|> Lit <$> integer <|> Var <$> try identifier
```

A **term** is one or more atoms, which we fold into left-associative application:

```haskell
term :: Parser Expr
term = foldl1 App <$> some atom
```

`some atom` greedily consumes as many atoms as it can find. `foldl1 App` folds them left to right: `[f, x, y, z]` becomes `App (App (App f x) y) z`. Left-associativity is the standard for application – `f x y` means `(f x) y`, i.e. apply `f` to `x` first, then apply the result to `y`. This is how currying works.

Some examples of how application interacts with the operator table:

```
f x + 1      →  App (Var "f") (Var "x")  parsed as term, then + 1 added
               → Add (App (Var "f") (Var "x")) (Lit 1)

1 + f x      →  1 + (f x)  since term = some atom, f x is one term
               → Add (Lit 1) (App (Var "f") (Var "x"))
```

Application binds tighter than any binary operator, which is what we expect: `f x + 1` means `(f x) + 1`, not `f (x + 1)`.

### The `try` problem

There's a subtlety hiding in `Var <$> try identifier`. Why `try`?

`some atom` works by repeatedly trying to parse another atom. It stops when `atom` fails – but in megaparsec, there's a distinction between failing _without consuming input_ (backtrackable) and failing _after consuming input_ (committed). Once a parser consumes input and then fails, megaparsec treats it as a hard error and won't try alternatives.

Consider `let f = x in f 5`. After parsing `x`, the parser doesn't know if `x` is going to be applied to an argument; `some atom` tries one more atom on `in f 5`. The `atom` parser tries each alternative in order:

1. `parens expr` – fails immediately on `i` (not `(`), no input consumed.
2. `Lit <$> integer` – fails immediately on `i` (not a digit), no input consumed.
3. `Var <$> identifier` – `letterChar` consumes `i`, `many alphaNumChar` consumes `n`. Now `name = "in"`, a keyword which can't be an identifier. The parser fails... but it has already consumed `i` and `n`.

At that point, `some atom` sees `atom` fail after consuming input. It can't backtrack to before the `i` and `n`. The error propagates upward, crashing out of the whole expression – and the outer `letExpr` never gets its `keyword "in"`.

The fix is `try identifier`: `try` wraps a parser so that if it fails (even after consuming input), it rewinds the input as if nothing was consumed. When `try identifier` encounters `in` and fails the keyword check, it restores the input position to before `i`. `atom` then fails without consuming anything, `some atom` stops cleanly, and `letExpr` can proceed to consume `in`.

Notice we only use `try` in `atom`, not in the `letExpr` and `lambdaExpr` parsers where `identifier` is called directly. Those parsers have already committed (they've consumed `let` or `\`), so a keyword-as-identifier error there is a genuine problem that should be reported. In `atom`, a keyword just means "not a variable reference here, stop the application chain."

## A new kind of value

Right now, `eval` returns `Either String Int`. With functions in the language, the evaluator needs to return something that can be either an integer or a function. We introduce a `Val` type:

```haskell
-- src/Helium/Eval.hs
data Val
  = VInt Int
  | VClosure String Expr Env
  deriving (Show)

type Env = Map String Val
```

`VInt` wraps an integer. `VClosure` represents a function – it holds three things:

- The **parameter name** (`String`)
- The **body** (`Expr`) – the expression to evaluate when the function is called
- The **captured environment** (`Env`) – a snapshot of the bindings visible when the function was defined

Notice that `Env` is `Map String Val`, and `Val` contains `VClosure ... Env`. This is a recursive type: the environment maps names to values, and values can be closures that contain environments. Haskell handles mutually recursive type definitions like this without any special syntax.

The previous `type Env = Map String Int` becomes `type Env = Map String Val`. Every occurrence of `Int` in the evaluator that represents a _value_ now becomes `Val`.

## Closures

Before writing the evaluator cases for `Lam` and `App`, it's worth understanding what closures are and why they matter.

Consider:

```
let x = 10 in
  let f = \y -> x + y in
    let x = 99 in
      f 1
```

What should `f 1` return? If `f` sees the current `x = 99`, you'd get `100`. But Helium uses **lexical scoping**: A function sees the bindings from where it was _defined_, not where it was _called_. So `f` was defined when `x = 10`, meaning `f 1` returns `11`.

The mechanism is the closure: when we create `VClosure "y" (Add (Var "x") (Var "y")) env`, we snapshot the current environment into the closure. Later, when we call `f`, we use _that_ environment (where `x = 10`), not the call-site environment (where `x = 99`).

**A closure is just a function body paired with the environment it was born in**.

## Evaluating functions

The `Lam` case is trivial – just wrap everything into a `VClosure`:

```haskell
  Lam parameter body -> Right $ VClosure parameter body env
```

The current `env` is captured into the closure at definition time.

`App` is where the action is:

```haskell
  App fExpr argExpr -> do
    f <- eval env fExpr
    case f of
      VClosure param body closureEnv -> do
        arg <- eval env argExpr
        eval (Map.insert param arg closureEnv) body
      val -> Left $ "Expected closure, got " <> show val
```

Step by step:

1. Evaluate `fExpr` to get the function value `f`.
2. Check that `f` is actually a closure – if not, report an error including what we got.
3. Evaluate `argExpr` in the _current_ environment to get the argument.
4. Evaluate the closure's `body` in `closureEnv` extended with the argument binding.

The order here is deliberate: we check that `f` is callable _before_ evaluating the argument. There's no point doing work on the argument if we're about to error on the function.

Note that in step 4, we use `closureEnv` (the closure's environment), not `env` (the call-site environment). This is what makes lexical scoping work. `closureEnv` is the environment where `f` was _defined_; we extend it with the parameter binding and evaluate the body inside that. The call-site's bindings are invisible to `f`'s body.

## Arithmetic with values

Arithmetic now has to deal with the fact that values might not be integers. We extract an `Int` from a `Val` using `asInt`:

```haskell
asInt :: Val -> Either String Int
asInt (VInt x) = Right x
asInt val      = Left $ "Expected integer, got " <> show val
```

Matching only on `VInt` and passing anything else to the wildcard case means if someone writes `(\x -> x) + 1`, `asInt` will report `"Expected integer, got VClosure "x" (Var "x") (fromList [])"` – you can see exactly what value caused the problem.

The `binOp` helper uses `asInt` to unwrap both sides before applying the operation:

```haskell
binOp :: Env -> (Int -> Int -> Int) -> Expr -> Expr -> Either String Val
binOp env op a b = do
  a' <- eval env a >>= asInt
  b' <- eval env b >>= asInt
  Right . VInt $ op a' b'
```

Each arithmetic case in the evaluator now goes through `binOp`, and `Neg` uses `asInt` directly:

```haskell
  Neg a   -> VInt . negate <$> (eval env a >>= asInt)
  -- ...
  Div a b -> do
    x <- eval env a >>= asInt
    y <- eval env b >>= asInt
    if y == 0 then Left "Division by 0" else Right . VInt $ x `div` y
```

## The REPL

The REPL is unchanged. `eval` now returns `Either String Val` instead of `Either String Int`, but `show result` handles `Val` via the derived `Show` instance – results print as `VInt 7` rather than `7`, and closures print their full internal representation, which is useful while we're building intuition for how the evaluator works.

## Trying it out

A basic lambda and application:

```
h> let double = \x -> x + 2 in double 5
Let "double" (Lam "x" (Add (Var "x") (Lit 2))) (App (Var "double") (Lit 5))
VInt 7
```

Currying – multi-argument functions via nested lambdas:

```
h> let add = \x -> \y -> x + y in add 3 4
Let "add" (Lam "x" (Lam "y" (Add (Var "x") (Var "y")))) (App (App (Var "add") (Lit 3)) (Lit 4))
VInt 7
```

`add 3` partially applies `add` – it evaluates to `VClosure "y" (Add (Var "x") (Var "y")) (fromList [("x",VInt 3)])`. The closure has captured `x = 3` in its environment. Then `add 3 4` applies that closure to `4`, substituting `y = 4` and evaluating `x + y = 7`.

Higher-order functions – `compose` takes two functions and returns their composition:

```
h> let compose = \f -> \g -> \x -> f (g x) in let double = \x -> x * 2 in let inc = \x -> x + 1 in compose double inc 5
...
VInt 12
```

`compose double inc` returns a new closure that applies `inc` first, then `double`. Passing it `5` gives `double (inc 5) = double 6 = 12`. `compose` itself takes three arguments via currying. `compose double inc` is two partial applications before the final `5`.

Lexical scoping – the closure captures `x = 10`, even after `x` is rebound to `99` in the call-site scope:

```
h> let x = 10 in let f = \y -> x + y in let x = 99 in f 1
Let "x" (Lit 10) (Let "f" (Lam "y" (Add (Var "x") (Var "y"))) (Let "x" (Lit 99) (App (Var "f") (Lit 1))))
VInt 11
```

The result is `11`, not `100`. `f`'s closure was created when `x = 10`, so that's the `x` it sees.

Applying a non-function:

```
h> 1 2
App (Lit 1) (Lit 2)
Runtime Error: Expected closure, got VInt 1
```

Arithmetic on a function:

```
h> (\x -> x) + 1
Add (Lam "x" (Var "x")) (Lit 1)
Runtime Error: Expected integer, got VClosure "x" (Var "x") (fromList [])
```

Both errors show the actual value that caused the problem – useful when values are more complex than these toy examples. Expressions from chapters 1 and 2 still work unchanged.

Helium is now a dynamically typed functional language. The evaluator can fail with type errors at runtime – "Expected integer, got VClosure..." – but it can't catch them before running.

In the next chapter, we'll add type annotations and a type checker. Instead of finding out at runtime that you applied `+` to a function, the checker will catch it statically, before evaluation begins. It's the first step toward making Helium's type system real.
