# Chapter 4b: GADTs vs System F

In Chapter 4, we built an elaborator that translates surface `Expr` into a `Core` IR, and an evaluator that trusts the elaborator. But the evaluator still pattern-matches on value constructors and uses `error "impossible"` for branches the type checker has ruled out. If we _know_ the types are correct, why do those branches exist at all?

> [!NOTE]
> The GADT-based implementation is on the [`chapter/004b`](https://github.com/dbalmain/helium-lang/tree/chapter/004b) branch.

This side chapter explores the question by comparing two approaches to type-safe intermediate representations:

> **[GADT](https://en.wikipedia.org/wiki/Generalized_algebraic_data_type)** (Generalized Algebraic Data Type): A Haskell language extension where each constructor of a data type can specify its own return type. A regular data type `data Expr = LitI Int | LitB Bool` gives every constructor the same type (`Expr`). A GADT `data Expr a where LitI :: Int -> Expr Int; LitB :: Bool -> Expr Bool` lets `LitI` produce `Expr Int` and `LitB` produce `Expr Bool`. When you pattern-match on a constructor, GHC learns what the type parameter is.

> **[System F](https://en.wikipedia.org/wiki/System_F)**: A typed lambda calculus where types are carried as explicit data rather than encoded in the host language's type system. In a System F-style compiler, the intermediate representation is a plain data type annotated with type information. The elaborator checks and attaches types; a separate lint pass can verify them independently. GHC's own Core is a variant of System F.

1. **The GADT approach** encodes types in Haskell's type system. `Core a` is parameterised by its result type, so `CLit :: Int -> Core Int` and `CApp :: Core (a -> b) -> Core a -> Core b`. The evaluator becomes `eval :: Core a -> Val a`, and ill-typed Core is unrepresentable.

2. **The System F approach** (what we used in Chapter 4) carries types as data in a plain ADT. The elaborator verifies types and produces `Core`, but Haskell can't see those guarantees. The evaluator trusts the elaborator, and `lint` provides independent verification.

Both approaches eliminate type errors from the evaluator. They differ in _where_ the guarantee lives and what trade-offs follow from that choice.

> [!NOTE]
> If you're unfamiliar with some of these concepts, you may want to skip to the [tutorial](#tutorial-try-both-approaches-yourself) at the end.

## The GADT approach

With GADTs, Core's type parameter tracks what each expression evaluates to:

```haskell
data Core a where
  CLit :: Int -> Core Int
  CVar :: String -> Core a
  CLet :: String -> Core b -> Core a -> Core a
  CLam :: SType a -> String -> Core b -> Core (a -> b)
  CApp :: Core (a -> b) -> Core a -> Core b
  CNeg :: Core Int -> Core Int
  CAdd :: Core Int -> Core Int -> Core Int
  ...
```

`CAdd` can only hold `Core Int` children. You can't construct `CAdd (CLam ...)` because the types don't unify. GHC enforces this at compile time.

The evaluator is _nearly_ total (no `error "impossible"` for type mismatches):

```haskell
eval :: Env -> Core a -> Either String (Val a)
```

If the input is `Core Int`, the output is `Val Int`. GHC knows this from the constructor, so every branch returns the right type without runtime checks.

### The cost

The type parameter creates friction whenever you don't know the type statically. A parser produces expressions of unknown type, so you need existential wrappers:

```haskell
data SomeCore where
  SomeCore :: SType a -> Core a -> SomeCore
```

`SType` is a singleton that mirrors the type parameter as a runtime value. You need it to recover type information after unwrapping the existential. The elaborator returns `Either String SomeCore` and must thread singletons through every case.

The environment is a `Map String SomeVal`, which means variable lookup requires `unsafeCoerce`:

```haskell
eval env (CVar name) =
  case env Map.! name of
    SomeVal v -> Right (unsafeCoerce v)
```

The elaborator proved both that the variable is bound and that it has the right type, but GHC can't verify either. The `Map.!` is partial (it crashes if the key is missing), and the `unsafeCoerce` trusts that the stored value has the expected type. This is the GADT approach's dirty secret: you still need trust _somewhere_, and it ends up here.

### The real problem: transformations

Consider a constant-folding pass. With the GADT, its signature is:

```haskell
optimize :: Core a -> Core a
```

This is type-safe by construction. If you accidentally return a `CLit` where a `CLam` was expected, GHC catches it. No lint needed.

But the type parameter helps only when input and output types match. Anywhere you need to store or collect values of different types, you need existential wrappers — types that hide the parameter behind a constructor like `SomeVal`. The environment is a good example: it can't be `Map String (Val a)` because different variables have different types. It has to be `Map String SomeVal`, and every lookup requires unwrapping and trusting the type with `unsafeCoerce`. The same pattern repeats for any data structure or pass that handles Core nodes of mixed types. What's straightforward with a plain ADT becomes a careful dance with wrappers. You start to see this even with our simple language. While the GADT-based evaluator is shorter, there is a lot more code to make that possible.

## The System F approach

With a plain ADT, Core-to-Core transformations are just `Core -> Core`. Subexpressions go in `[Core]`. No existentials, no singletons, no `unsafeCoerce`. The cost is that you can construct ill-typed Core, and the evaluator has `error "impossible"` branches. `lint` catches these if they ever happen.

This is the trade-off real compilers make. GHC's own Core is a plain ADT with a lint pass, not a GADT. The engineering argument: every optimization pass would need to satisfy GADT constraints, making them much harder to write. Core Lint gives the same safety guarantee at lower engineering cost.

## Why we chose System F for Helium

Helium will eventually have optimization passes, and keeping those simple matters more than eliminating `error "impossible"` from the evaluator. The System F approach also maps more directly to what a non-Haskell implementation would do (relevant since we'll rewrite in Rust).

The GADT approach is worth understanding because it shows what "types as propositions" looks like in practice. Singletons, existentials, and type equality proofs (`Data.Type.Equality`) are concepts that come up in advanced Haskell programming beyond compilers.

## Tutorial: try both approaches yourself

To get a hands-on feel for the trade-offs, there's a self-contained tutorial with exercises for both approaches. It uses a simpler expression language (Int and Bool literals, Add, Eq, If) so you can focus on the GADT machinery without the complexity of lambdas and closures.

Clone the tutorial branch:

```
cd helium-book
git fetch origin tutorial/gadts-vs-systemf
git checkout tutorial/gadts-vs-systemf
```

You'll find:

| File                  | What it covers                                                                                                            |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `GADTs.hs`            | Build up from untyped eval to GADTs, singletons, existentials, type equality proofs, elaboration, and typed optimizations |
| `SystemF.hs`          | The same problem solved with types-as-data: elaboration, a trusting evaluator, Core Lint, and plain ADT transformations   |
| `GADTs-solution.hs`   | Reference solutions for the GADT exercises                                                                                |
| `SystemF-solution.hs` | Reference solutions for the System F exercises                                                                            |

Each file is standalone. Replace every `error "todo"` with a working implementation, then run with `runghc GADTs.hs` (or `runghc SystemF.hs`). Tests at the bottom tell you what's passing.

If you have nix with flakes enabled, `nix develop` sets up GHC and HLS.

Working through both `GADTs.hs` and `SystemF.hs` should give you a good feel for the contrast between each approach.
