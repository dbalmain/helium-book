# Chapter 4: Simply-Typed Checking

In Chapter 3, errors like `1 2` or `(\x -> x) + 1` crashed at runtime. The evaluator dutifully ran the code, hit a type mismatch, and reported an error. Now, we'll catch these mistakes _before_ the code runs, by adding a type checker that walks the AST and verifies that types line up.

> [!NOTE]
> The complete code for this chapter is on the [`chapter/004`](https://github.com/dbalmain/helium-lang/tree/chapter/004) branch.

This chapter introduces:

1. A **Type** data type (`TInt` and `TFun`) to represent the types of expressions.
2. **Mandatory type annotations** on lambda parameters: `\(x : Int) -> body` instead of `\x -> body`.
3. A **typed intermediate representation** (`Core`) that the elaborator produces and the evaluator consumes.
4. An **elaborator** that translates the surface AST into `Core`, catching type errors along the way.

No type inference yet. Every lambda parameter must be annotated. That's a strict requirement we'll relax in Chapter 6, but it keeps the type checker simple: we never have to _guess_ a type, only _check_ that the types we're given are consistent.

## Types as a data type

What types exist in Helium? So far, just two kinds:

```haskell
-- src/Helium/Syntax.hs
data Type
  = TInt
  | TFun Type Type
  deriving (Show, Eq)
```

`TInt` is the type of integers. `TFun a b` is the type of a function that takes an `a` and returns a `b`. A function `Int -> Int` is `TFun TInt TInt`. A curried two-argument function `Int -> Int -> Int` is `TFun TInt (TFun TInt TInt)` – function types are right-associative, so `a -> b -> c` means `a -> (b -> c)`.

That's it. Two constructors cover every type in the language. Because `TFun`'s argument and return types are themselves `Type` values, we can represent arbitrarily nested function types like `(Int -> Int) -> Int -> Int`.

We derive `Eq` because the type checker needs to compare types for equality. When we check that an argument has the right type for a function parameter, we'll use `==` on `Type` values.

## Annotated lambdas

In Chapter 3, lambdas looked like `\x -> body` and the AST was `Lam String Expr`. Without type annotations, the type checker would have to _infer_ the parameter type from how it's used. That's coming in chapter 6 with Hindley-Milner inference.

For now, we'll require annotations: `\(x : Int) -> body`. The AST gains a `Type` field:

```haskell
data Expr
  = Lit Int
  | Var String
  | Let String Expr Expr
  | Lam String Type Expr       -- \(x : Int) -> body
  | App Expr Expr
  | Neg Expr
  | Add Expr Expr
  | Sub Expr Expr
  | Mul Expr Expr
  | Div Expr Expr
  deriving (Show, Eq)
```

`Lam "x" TInt body` represents `\(x : Int) -> body`. The type is mandatory; there's no `Maybe Type` here. Every lambda _must_ declare the type of its parameter.

> [!Note]
> Why the parentheses around `(x : Int)`? Consider what `\x : Int -> Int -> body` would mean without them. Is `Int -> Int` a function type annotation, or is `Int` the annotation and `-> body` the body? The parens make it unambiguous: `\(x : Int -> Int) -> body` is clearly a lambda taking a function-typed parameter. There are other options we could consider, such as putting the parentheses around the type only, or using a different operator in the type, but we're keeping the syntax close to Haskell for now.

## Parsing types

We need a parser for types. Type syntax has two levels, mirroring how we split expression parsing into atoms and compound expressions.

A **type atom** is either `Int` or a parenthesised type expression:

```haskell
-- src/Helium/Parser.hs
typeAtom :: Parser Type
typeAtom = TInt <$ keyword "Int" <|> parens typeExpr
```

`TInt <$ keyword "Int"` parses the literal text `Int` (followed by a non-alphanumeric boundary) and produces `TInt`. The `<$` operator discards the parsed value and replaces it with the left-hand side.

Note that `Int` is _not_ added to the keywords list. Keywords like `let` and `in` are syntactic constructs that must be distinguished from identifiers everywhere. `Int` is a type name. It only appears in type positions (after `:` in a lambda parameter), where the parser calls `typeAtom` instead of `identifier`. There's no ambiguity to resolve. If someone writes `let Int = 5 in Int + 1`, that's fine. `Int` as a variable name lives in expression positions, `Int` as a type lives in type positions, and the parser handles each context separately.

A **type expression** is one or more type atoms separated by `->`, folded right-associatively:

```haskell
typeExpr :: Parser Type
typeExpr = foldr1 TFun <$> sepBy1 typeAtom (symbol "->")
```

`sepBy1` parses one or more type atoms separated by `->`. `foldr1 TFun` folds them into a right-associative chain: `[TInt, TInt, TInt]` becomes `TFun TInt (TFun TInt TInt)`. This matches the standard convention that `Int -> Int -> Int` means `Int -> (Int -> Int)`.

## Parsing annotated lambdas

The lambda parser changes from `\x -> body` to `\(x : Int) -> body`:

```haskell
lambdaExpr :: Parser Expr
lambdaExpr = do
  _ <- symbol "\\"
  _ <- symbol "("
  parameter <- identifier <?> "parameter"
  _ <- symbol ":"
  ty <- typeExpr
  _ <- symbol ")"
  _ <- symbol "->"
  Lam parameter ty <$> expr
```

## Checking vs. elaboration

The simplest type checker would walk the AST, verify that types are consistent, and return the expression's type as `check :: Expr -> Either String Type`. That would work, but it wastes information. The type checker figures out the type of every subexpression, then throws it all away. The evaluator receives the same untyped AST and has to handle "impossible" cases. We'd still be checking someone isn't trying to subtract a closure from an integer, even though the type checker already proved that can't happen.

A better approach: instead of returning just a type, the type checker produces a **new, typed AST** where every node carries its type. This is called **elaboration**: translating from surface syntax to a typed intermediate representation. The evaluator then operates on this typed intermediate representation (IR) and can trust that types are correct, eliminating entire categories of runtime errors.

This is how most real compilers work. GHC, for example, has separate representations for pre-typecheck and post-typecheck syntax. The type checker is a translation pass from one to the other.

## A typed intermediate representation

The typed IR lives in a new module, `Helium.Core`. It's structurally almost identical to `Expr`:

```haskell
-- src/Helium/Core.hs
data Core
  = CLit Int
  | CVar String
  | CLet String Core Core
  | CLam Type String Core    -- carries param type
  | CApp Core Core
  | CNeg Core
  | CAdd Core Core
  | CSub Core Core
  | CMul Core Core
  | CDiv Core Core
  deriving (Show)
```

Notice what's different from `Expr`:

- The constructors are prefixed with `C` to avoid clashing with `Expr`'s constructors.
- `CLam` carries the parameter's `Type`. In `Expr`, the `Lam` constructor already had this annotation from the parser. `Core` preserves it so the type information survives into the IR, which is useful for debugging, pretty-printing, and the `lint` function we'll see shortly.

Not much has changed. These types will further diverge as we add features like conditionals and pattern matching.

> [!Note]
> I've made the choice to keep return types out of Core. The elaborator returns a `(Type, Core)` pair with the top-level type alongside the Core tree. It's really this tuple that represents the typed IR, rather than just Core.
>
> Individual node types can be recomputed by the `lint` function if needed. This allows us to pass a lighter Core to the evaluator which doesn't care about return types. We'll see as we go if this was the right choice.

## The elaborator

The elaborator lives in `Helium.Check`. Its job: walk the surface `Expr`, verify types, and build a `Core` tree.

```haskell
-- src/Helium/Check.hs
module Helium.Check (elaborate) where

import Data.Map.Strict (Map)
import Data.Map.Strict qualified as Map
import Helium.Core (Core (..))
import Helium.Syntax (Expr (..), Type (..))

type TEnv = Map String Type
```

`TEnv` is a _type environment_: it maps variable names to their types. It plays the same role as `Env` in the evaluator, but tracks types instead of values.

### The `expect` helper

A small helper for type equality:

```haskell
expect :: Type -> (Type, Core) -> Either String Core
expect expected (actual, core)
  | expected == actual = Right core
  | otherwise = Left $ "Expected " <> show expected <> ", got " <> show actual
```

`expect` takes an expected type and a `(Type, Core)` pair (the result of elaborating a subexpression). If the types match, it returns the `Core`; otherwise it reports an error. This signature lets us pipe `elab` results directly into `expect`:

```haskell
aCore <- elab env a >>= expect TInt
```

This is the simplest possible version – just `==` on types. In Chapter 6, when we add type variables and inference, `expect` will become _unification_, a much more interesting operation. But for now, equality is all we need.

### Walking the AST

The core of the elaborator is `elab`, which takes a type environment and an expression, and returns a type alongside a `Core` node:

```haskell
elab :: TEnv -> Expr -> Either String (Type, Core)
elab env expr = case expr of
```

**Literals** produce `TInt` and a `CLit`:

```haskell
  Lit x -> Right (TInt, CLit x)
```

**Variables** are looked up in the type environment:

```haskell
  Var name -> case Map.lookup name env of
    Just ty -> Right (ty, CVar name)
    Nothing -> Left $ "Unbound variable: " <> name
```

This variable check means that we know there will be a bound variable in any code that inherits this environment, and what the type of the variable will be. We won't need to reverify in the evaluator.

**Let bindings** elaborate the definition, add its type to the environment, then elaborate the body. The whole let expression has the body's type:

```haskell
  Let name def body -> do
    (defTy, defCore) <- elab env def
    (bodyTy, bodyCore) <- elab (Map.insert name defTy env) body
    Right (bodyTy, CLet name defCore bodyCore)
```

Each call to `elab` returns a `(Type, Core)` pair. We destructure both, use the definition's type to extend the environment, and return the body's type as the overall type.

**Lambdas** are, for now, the only construct where the user has a say in the types:

```haskell
  Lam param ty body -> do
    (bodyTy, bodyCore) <- elab (Map.insert param ty env) body
    Right (TFun ty bodyTy, CLam ty param bodyCore)
```

Add the parameter with its declared type to the environment, elaborate the body, and return `TFun paramTy bodyTy`. The `CLam` carries the parameter type so it's preserved in the IR.

**Application** is the most interesting case:

```haskell
  App f arg -> do
    (fTy, fCore) <- elab env f
    case fTy of
      TFun paramTy retTy -> do
        argCore <- elab env arg >>= expect paramTy
        Right (retTy, CApp fCore argCore)
      _ -> Left $ "Expected function type, got " <> show fTy
```

Elaborate the function, check that its type is `TFun paramTy retTy`, elaborate the argument, verify the argument's type matches `paramTy` using `expect`, and return `retTy`. If the function isn't a function type (like `1 2`, where `1` has type `TInt`), report an error.

The `elab env arg >>= expect paramTy` pipeline works because `elab` returns `Either String (Type, Core)` and `expect` takes `(Type, Core)`, so they chain together cleanly with `>>=`.

**Arithmetic** requires both operands to be `TInt` and produces `TInt`:

```haskell
  Neg a -> do
    aCore <- elab env a >>= expect TInt
    Right (TInt, CNeg aCore)

  Add a b -> elabBinOp env CAdd a b
  Sub a b -> elabBinOp env CSub a b
  Mul a b -> elabBinOp env CMul a b
  Div a b -> elabBinOp env CDiv a b

elabBinOp :: TEnv -> (Core -> Core -> Core) -> Expr -> Expr -> Either String (Type, Core)
elabBinOp env op a b = do
  aCore <- elab env a >>= expect TInt
  bCore <- elab env b >>= expect TInt
  Right (TInt, op aCore bCore)
```

`elabBinOp` takes the `Core` constructor as an argument (e.g. `CAdd`), elaborates both sides, checks they're `TInt` via `expect`, and returns the result as `TInt`.

### The entry point

```haskell
elaborate :: Expr -> Either String (Type, Core)
elaborate = elab Map.empty
```

Start with an empty type environment. Top-level expressions can't reference any variables that haven't been introduced by `let` or `Lam` within the expression itself.

## Lint: a safety net for the compiler

The evaluator trusts the elaborator. If elaboration succeeds, `eval` assumes every type is correct. But what if the elaborator has a bug? We'd get a confusing crash from an `error "impossible"` deep in the evaluator, with no indication of _what_ went wrong in the type system.

`lint` is our defence against that. It's an independent type checker that operates on `Core`, re-deriving the type of every node from scratch. If `lint` and the elaborator disagree, we've found a compiler bug.

```haskell
-- src/Helium/Core.hs
lint :: TEnv -> Core -> Either String Type
lint env core = case core of
  CLit _ -> Right TInt
  CVar name -> maybe (Left $ "lint: Unbound variable: " <> name) Right (Map.lookup name env)

  CLet name def body -> do
    defTy <- lint env def
    lint (Map.insert name defTy env) body

  CLam paramTy param body -> do
    bodyTy <- lint (Map.insert param paramTy env) body
    Right $ TFun paramTy bodyTy

  CApp f arg -> do
    fTy <- lint env f
    case fTy of
      TFun paramTy retTy -> do
        argTy <- lint env arg
        expect paramTy argTy
        Right retTy
      _ -> Left $ "lint: Expected function type, got " <> show fTy

  CNeg a -> do
    lint env a >>= expect TInt
    Right TInt

  CAdd a b -> lintBinOp env a b
  CSub a b -> lintBinOp env a b
  CMul a b -> lintBinOp env a b
  CDiv a b -> lintBinOp env a b
```

`lint` mirrors the elaborator's logic, but it's simpler – it only computes types, it doesn't build anything. The `CLam` case shows why we stored the parameter type on `Core`: without it, `lint` couldn't determine the lambda's type independently.

We prefix error messages with `"lint: "` so they're immediately distinguishable from elaboration errors. A lint error in normal use means the elaborator produced a `Core` tree that doesn't type-check, which is a compiler bug worth investigating.

We won't wire `lint` into the pipeline yet. It's there for development-time debugging, not as a user-facing pass. But it's a useful sanity check to run manually when changing the elaborator, and it becomes essential once we add optimization passes that transform Core (constant folding, inlining, etc.). Each transformation could introduce a type error, and running `lint` after each pass catches bugs in the transformation. GHC's `-dcore-lint` flag exists for exactly this reason.

## An evaluator that trusts the types

With elaboration, the evaluator operates on `Core` instead of `Expr`. Runtime type mismatches should now be impossible, so we can eliminate the error paths for them.

> [!NOTE]
> If we know the types are correct, why does the evaluator still pattern-match on `VInt` and `VClosure` at all? With Haskell's GADTs, we could encode the types directly into `Core`'s type parameter, making `eval :: Core a -> Val a` and eliminating the impossible branches entirely. [Chapter 4b](chapter/004b.md) explores this alternative and compares the trade-offs.

```haskell
-- src/Helium/Eval.hs
module Helium.Eval (eval, Val (..)) where

import Data.Map.Strict (Map)
import Data.Map.Strict qualified as Map
import Helium.Core (Core (..))

type Env = Map String Val

data Val
  = VInt Int
  | VClosure String Core Env
  deriving (Show)
```

`VClosure` now holds `Core` instead of `Expr`, since closures capture typed expressions.

`binOp` is a helper for integer binary operations:

```haskell
binOp :: Env -> (Int -> Int -> Int) -> Core -> Core -> Either String Val
binOp env op coreA coreB = do
  valA <- eval env coreA
  valB <- eval env coreB
  case (valA, valB) of
    (VInt a, VInt b) -> Right . VInt $ op a b
    _ -> error "impossible: type checker guarantees integer operands"
```

The naming convention makes the transformation pipeline visible: `coreA` (the `Core` node) becomes `valA` (the evaluated `Val`) becomes `a` (the unwrapped `Int`). The `error` call is a safety net, not error handling. If it ever fires, the elaborator has a bug.

The evaluator itself:

```haskell
eval :: Env -> Core -> Either String Val
eval env core = case core of
  CLit x -> Right $ VInt x
  CVar name -> Right $ env Map.! name
  CLet name coreDef coreBody -> do
    val <- eval env coreDef
    eval (Map.insert name val env) coreBody
  CLam _ty param body -> Right $ VClosure param body env
  CApp coreF coreArg -> do
    valF <- eval env coreF
    case valF of
      VClosure param body closureEnv -> do
        valArg <- eval env coreArg
        eval (Map.insert param valArg closureEnv) body
      _ -> error "impossible: type checker guarantees function application"
  CNeg coreA -> do
    valA <- eval env coreA
    case valA of
      VInt a -> Right . VInt $ negate a
      _ -> error "impossible: type checker guarantees integer operand"
  CAdd coreA coreB -> binOp env (+) coreA coreB
  CSub coreA coreB -> binOp env (-) coreA coreB
  CMul coreA coreB -> binOp env (*) coreA coreB
  CDiv coreA coreB -> do
    valA <- eval env coreA
    valB <- eval env coreB
    case (valA, valB) of
      (VInt a, VInt b) ->
        if b == 0 then Left "Division by 0" else Right . VInt $ a `div` b
      _ -> error "impossible: type checker guarantees integer operands"
```

Compare this to the Chapter 3 evaluator. There, `asInt` returned `Either String Int` because a type mismatch was a real possibility; the user might write `1 + (\x -> x)` and reach that path. Now that the elaborator has verified every type, the wrong-constructor cases are unreachable. The `error` calls are safety nets that would indicate a compiler bug, not a user mistake.

The only remaining `Left` is division by zero, a runtime error that our type system doesn't prevent. Even unbound variables are now impossible: the elaborator proved every variable is in scope, so `CVar` uses `Map.!` instead of a safe lookup.

> [!NOTE]
> Unchecked division by zero is a design choice I've made for simplicity (or perhaps familiarity?) in these early stages. [Pony](https://tutorial.ponylang.io/gotchas/divide-by-zero.html) makes division a partial operation that must be wrapped in a `try` block. [Idris](https://github.com/idris-lang/Idris-dev/blob/master/libs/contrib/Data/Nat/DivMod.idr) uses dependent types to require a compile-time proof that the divisor is nonzero. Either approach would eliminate this last runtime error. Something to revisit as Helium's error handling story develops.

## The updated pipeline

The REPL now runs three stages: parse, elaborate, evaluate.

```haskell
-- app/Main.hs
case parseExpr input of
  Left err -> outputStrLn $ "Parse Error: " <> err
  Right expr -> do
    outputStrLn $ show expr
    case elaborate expr of
      Left err -> outputStrLn $ "Type Error: " <> err
      Right (ty, core) -> do
        outputStrLn $ "Type: " <> show ty
        case eval mempty core of
          Left err -> outputStrLn $ "Runtime Error: " <> err
          Right result -> outputStrLn $ show result
```

On success, `elaborate` returns `(Type, Core)`. We destructure the pair to display the type and pass the `Core` tree to `eval`. The evaluator never sees the surface `Expr` at all.

## Trying it out

Arithmetic still works:

```
h> 1 + 2
Type: TInt
VInt 3
```

A typed lambda applied to an argument:

```
h> let double = \(x : Int) -> x + x in double 5
Type: TInt
VInt 10
```

A lambda without application returns a function type. Notice the closure now contains `Core` nodes rather than surface `Expr`:

```
h> \(x : Int) -> x + 1
Type: TFun TInt TInt
VClosure "x" (CAdd (CVar "x") (CLit 1)) (fromList [])
```

Curried functions work through nested annotations:

```
h> let add = \(x : Int) -> \(y : Int) -> x + y in add 3 4
Type: TInt
VInt 7
```

Type errors are caught before evaluation:

```
h> 1 2
Type Error: Expected function type, got TInt
```

```
h> (\(x : Int) -> x) + 1
Type Error: Expected TInt, got TFun TInt TInt
```

```
h> let f = \(x : Int) -> x in f f
Type Error: Expected TInt, got TFun TInt TInt
```

And division by zero is the only remaining runtime error:

```
h> 10 / 0
Type: TInt
Runtime Error: Division by 0
```

## What we built

| File            | What changed                                                                          |
| --------------- | ------------------------------------------------------------------------------------- |
| `Helium.Syntax` | Added `Type` data type; `Lam` now carries a mandatory type annotation                 |
| `Helium.Parser` | Added `typeAtom`, `typeExpr`; updated `lambdaExpr` for `\(x : Int) -> body` syntax    |
| `Helium.Core`   | **New** – typed IR with `lint` for independent verification                           |
| `Helium.Check`  | **New** – elaborator: `elaborate :: Expr -> Either String (Type, Core)`               |
| `Helium.Eval`   | Now operates on `Core`; runtime type checks replaced with `error "impossible"` guards |
| `Main`          | Pipeline: parse → elaborate → eval                                                    |

The elaborator mirrors the evaluator's structure (same tree walk, same environment threading) but computes types and builds a typed IR instead of computing values. This parallel structure isn't a coincidence; it's a common pattern in language implementation. You'll see it again in Chapter 6.

But there's a problem: our type errors don't tell you _where_ in the code the mistake is. `"Expected function type, got TInt"` is useful, but it doesn't point at the `1` in `1 2`. Megaparsec gives us precise parse errors for free. In the next chapter, we'll add source spans to the AST so the elaborator can do the same.
