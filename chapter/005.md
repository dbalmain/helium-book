# Chapter 5: Spans and Errors

So far, Megaparsec gives us precise parse errors pointing at the exact source location. Errors from later phases (elaboration and evaluation) only describe the problem; they can't tell you where it happened, because the AST carries no source locations. In this chapter we thread source spans through to the elaborator so type errors can look as good as parse errors.

> [!NOTE]
> The complete code for this chapter is on the [`chapter/005`](https://github.com/dbalmain/helium-lang/tree/chapter/005) branch.

This chapter:

1. Switches the REPL to isocline for multi-line editing.
2. Adds a `Span` to every `Expr` constructor.
3. Introduces a `TypeError` type that pairs a span with an error message.
4. Updates `elaborate` to return `Either TypeError (Type, Core)`.
5. Renders type errors with a line, column, source snippet, and pointer.

> [!NOTE]
> I'll keep this chapter short. After implementing it, the changes to `Parser.hs` in particular felt heavy. Every future syntax change will now need to wire up spans, and we're nowhere near the point where anyone will be writing long programs in Helium. The cost outweighs the benefit for now.
>
> Chapter 6 picks up from the isocline switch but drops the spans. If you're not interested in the span plumbing, grab the REPL change below and skip ahead.

## Multi-line editing in the REPL

To show source locations in errors, we want programs spanning multiple lines. Haskeline, our original readline choice, doesn't do multi-line editing out of the box. We could built it, but that's not where I want to spend time. We'll switch to [isocline](https://hackage.haskell.org/package/isocline), a lightweight readline alternative with multi-line editing out of the box: press Ctrl-Enter to insert a newline, use the arrow keys to move around the whole buffer, and hit Enter to submit.

isocline also supports syntax highlighting, which we may wire up in a later chapter. For now all we want is multi-line input.

The REPL loop drops back to plain `IO`, with no `InputT` transformer:

```haskell
-- app/Main.hs
import System.Console.Isocline

repl :: IO ()
repl = do
  minput <- readlineMaybe "he"
  case minput of
    Nothing -> putStrLn ""
    Just input -> process input >> repl

main :: IO ()
main = do
  putStrLn "Helium Chapter 5 - Type an expression (Ctrl-D to quit, Ctrl-Enter for newline)"
  setHistory "helium_history.txt" 200
  _ <- enableMultiline True
  repl
```

`readlineMaybe` returns `Nothing` on EOF (Ctrl-D) and otherwise yields the full multi-line string the user submitted. `enableMultiline True` is the key call: without it, Enter always submits and Ctrl-Enter does nothing. `setHistory` persists the last 200 entries across sessions.

## Spans

A span is the source range that produced a particular AST node. We store a start and an end `SourcePos`, both from Megaparsec. In a more advanced interpreter, we'd also store a reference to the source file, but for now a single REPL input is all we have.

```haskell
-- src/Helium/Syntax.hs
data Span = Span SourcePos SourcePos
  deriving (Show, Eq)
```

Every `Expr` constructor gains a `Span` as its first field. We add two small helpers: `spanOf` pulls the span back out of any `Expr`, and `spanStart` extracts the start line and column for error rendering.

```haskell
data Expr
  = Lit Span Int
  | Var Span String
  | Let Span String Expr Expr
  | Lam Span String Type Expr
  | App Span Expr Expr
  | Neg Span Expr
  | Add Span Expr Expr
  | Sub Span Expr Expr
  | Mul Span Expr Expr
  | Div Span Expr Expr
  deriving (Show, Eq)

spanOf :: Expr -> Span
spanOf (Lit s _) = s
spanOf (Var s _) = s
spanOf (Let s _ _ _) = s
spanOf (Lam s _ _ _) = s
spanOf (App s _ _) = s
spanOf (Neg s _) = s
spanOf (Add s _ _) = s
spanOf (Sub s _ _) = s
spanOf (Mul s _ _) = s
spanOf (Div s _ _) = s

spanStart :: Span -> (Int, Int)
spanStart (Span pos _) = (unPos $ sourceLine pos, unPos $ sourceColumn pos)
```

We also introduce a `TypeError` that packages a span with an error message:

```haskell
data TypeError = TypeError Span String
```

`elaborate` will return `Either TypeError (Type, Core)` instead of `Either String (Type, Core)`.

`Core` is deliberately left alone. Once we've checked the program, the evaluator has no use for source locations, and the less we thread spans into Core, the less churn we incur later.

## Adding spans during parsing

Every `Expr` node now needs a span. The core helper is `spanned`, which wraps any parser and returns the span of the consumed input alongside the parsed value:

```haskell
-- src/Helium/Parser.hs
spanned :: Parser a -> Parser (Span, a)
spanned p = do
  start <- getSourcePos
  result <- p
  end <- getSourcePos
  pure (Span start end, result)
```

For atoms, `spanned` wraps the whole literal or identifier parser, and we build the node with `uncurry`:

```haskell
atom :: Parser Expr
atom =
  parens expr
    <|> (uncurry Lit <$> spanned integer)
    <|> (uncurry Var <$> spanned (try identifier))
```

Application spans can't use `spanned`. By the time `foldl1` folds two atoms into an `App`, we've already left the parser context where `getSourcePos` would be useful. Instead, we derive the application's span from its children, taking the start from the function and the end from the argument:

```haskell
term :: Parser Expr
term = foldl1 mkApp <$> some atom
  where
    mkApp f a =
      let Span start _ = spanOf f
          Span _ end = spanOf a
       in App (Span start end) f a
```

The operator table uses the same trick. A prefix operator remembers the position just before it consumes its symbol and closes the span at the end of the operand. A binary operator takes both ends from its operands:

```haskell
prefix :: String -> (Span -> Expr -> Expr) -> Operator Parser Expr
prefix name f = Prefix $ do
  start <- getSourcePos
  _ <- symbol name
  pure $ \a -> f (Span start (spanEnd (spanOf a))) a
  where
    spanEnd (Span _ e) = e

binary :: String -> (Span -> Expr -> Expr -> Expr) -> Operator Parser Expr
binary name f = InfixL $ do
  _ <- symbol name
  pure $ \a b ->
    let Span start _ = spanOf a
        Span _ end = spanOf b
     in f (Span start end) a b
```

`lambdaExpr` and `letExpr` each parse three payload values, so we factor out a small helper to avoid repeating the `spanned` boilerplate:

```haskell
withSpan :: (Span -> a -> b -> c -> Expr) -> Parser (a, b, c) -> Parser Expr
withSpan ex p = do
  (span_, (x, y, z)) <- spanned p
  pure $ ex span_ x y z

lambdaExpr :: Parser Expr
lambdaExpr = withSpan Lam $ do
  _ <- symbol "\\"
  _ <- symbol "("
  parameter <- identifier <?> "parameter"
  _ <- symbol ":"
  ty <- typeExpr
  _ <- symbol ")"
  _ <- symbol "->"
  body <- expr
  pure (parameter, ty, body)

letExpr :: Parser Expr
letExpr = withSpan Let $ do
  keyword "let"
  name <- identifier <?> "identifier"
  _ <- symbol "="
  definition <- expr
  keyword "in"
  body <- expr
  pure (name, definition, body)
```

This is the bulk of the span plumbing, and it's the main reason I've decided the feature isn't paying for itself yet. Every new syntactic construct will need the same treatment.

## Enriching type errors

With spans available on every `Expr`, the elaborator can blame a specific location when something goes wrong. `expect` now takes the span that should be reported on a mismatch:

```haskell
-- src/Helium/Check.hs
expect :: Span -> Type -> (Type, Core) -> Either TypeError Core
expect s expected (actual, core)
  | expected == actual = Right core
  | otherwise = Left $ TypeError s $ "Expected " <> show expected <> ", got " <> show actual
```

Call sites pass `spanOf a`, the span of the offending subexpression, not the span of the enclosing node. The error should point at the value with the wrong type, not at the operator above it:

```haskell
elab env expr = case expr of
  Lit _ x -> Right (TInt, CLit x)
  Var s name -> case Map.lookup name env of
    Just ty -> Right (ty, CVar name)
    Nothing -> Left $ TypeError s $ "Unbound variable: " <> name
  ...
  App s f arg -> do
    (fTy, fCore) <- elab env f
    case fTy of
      TFun paramTy retTy -> do
        argCore <- elab env arg >>= expect (spanOf arg) paramTy
        Right (retTy, CApp fCore argCore)
      _ -> Left $ TypeError s $ "Expected function type, got " <> show fTy
  Neg _ a -> do
    aCore <- elab env a >>= expect (spanOf a) TInt
    Right (TInt, CNeg aCore)
  Add _ a b -> elabBinOp env CAdd a b
  ...
```

There are two kinds of error sites in `App`. If the function's type isn't `TFun` at all, we blame `s`, the span of the whole application. If the argument's type is wrong, we blame `spanOf arg`, because the argument is the value that's out of place.

The binary operator helper drops the outer span entirely. Both operands carry their own spans, and each is checked against `TInt`:

```haskell
elabBinOp :: TEnv -> (Core -> Core -> Core) -> Expr -> Expr -> Either TypeError (Type, Core)
elabBinOp env op a b = do
  aCore <- elab env a >>= expect (spanOf a) TInt
  bCore <- elab env b >>= expect (spanOf b) TInt
  Right (TInt, op aCore bCore)
```

## Showing type errors

The REPL owns the rendering. When elaboration fails, `Main.hs` formats the `TypeError` against the original input: a header with line and column, the offending source line, a caret pointing at the column, and the error message.

```haskell
-- app/Main.hs
renderTypeError :: String -> TypeError -> String
renderTypeError source (TypeError span_ msg) =
  let (ln, col) = spanStart span_
      sourceLines = lines source
      lineText = fromMaybe "" (sourceLines !? (ln - 1))
      pointer = replicate (col - 1) ' ' <> "^"
      header = "Type Error at " <> show ln <> ":" <> show col <> ":"
      lineNumber = show ln
      divider = " | "
   in intercalate
        "\n   "
        [ header,
          lineNumber <> divider <> lineText,
          replicate (length lineNumber) ' ' <> divider <> pointer,
          msg
        ]
```

`!?` is the safe list indexer from `Data.List`. We wrap it with `fromMaybe ""` so a bad line number falls back to an empty source line instead of crashing the REPL. Line numbers are 1-based and list indices are 0-based, which is why we subtract one.

`intercalate "\n   "` joins the four lines with a newline and a 3-space indent, so the output is visually offset from the REPL prompt.

The pipeline wires it up:

```haskell
process :: String -> IO ()
process input =
  case parseExpr input of
    Left err -> putStrLn $ "Parse Error: " <> err
    Right expr -> do
      case elaborate expr of
        Left err -> putStrLn $ renderTypeError input err
        ...
```

We pass the original `input` string to `renderTypeError` so it can look up the offending source line.

## Trying it out

Type errors now point at the exact subexpression that's wrong:

```
he> let f = \(x : Int) -> x in f f
Type Error at 1:30:
   1 | let f = \(x : Int) -> x in f f
     |                              ^
   Expected TInt, got TFun TInt TInt
```

The caret lands on the second `f`: the argument, not the function. That's the value with the wrong type.

Multi-line input works too. Press Ctrl-Enter to insert a newline, use the arrow keys to move around the buffer, and hit Enter to submit:

```
he> let double = \(x : Int) -> x + x
    in double 5
Type: TInt
VInt 10
```

Errors on the second line get the line number you'd expect:

```
he> let f = \(x : Int) -> x
    in f f
Type Error at 2:8:
   2 | in f f
     |      ^
   Expected TInt, got TFun TInt TInt
```
