# Chapter 2: Variables and Let Bindings

By the end of this chapter, we'll be able to evaluate expressions like `let x = 5 in x + 1`. Before we get to variables though, we have a problem to fix: the evaluator can't report errors.

> [!NOTE]
> The complete code for this chapter is on the [`chapter/002`](https://github.com/dbalmain/helium-lang/tree/chapter/002) branch.

Right now, `eval` has the type `Expr -> Int`. It always returns a number, which means it has no way to say "something went wrong." Division by zero crashes the program, albeit with a pretty nice error from Haskell. When we add variables, we'll have another failure mode: looking up a variable that doesn't exist. We want to handle these cases gracefully.

## Error Handling with Either

Here's the updated evaluator:

```haskell
-- src/Helium/Eval.hs
module Helium.Eval (eval) where

import Helium.Syntax (Expr (..))

binOp :: (Int -> Int -> Int) -> Expr -> Expr -> Either String Int
binOp f a b = f <$> eval a <*> eval b

eval :: Expr -> Either String Int
eval expr = case expr of
  (Lit n) -> Right n
  (Neg e) -> negate <$> eval e
  (Add a b) -> binOp (+) a b
  (Sub a b) -> binOp (-) a b
  (Mul a b) -> binOp (*) a b
  (Div a b) -> do
    x <- eval a
    y <- eval b
    if y == 0 then Left "Division by 0" else Right $ x `div` y
```

The key change: `eval` now returns `Either String Int` instead of `Int`. `Right 7` means success, `Left "Division by 0"` means failure. This lets us use `<$>` and `<*>` to propagate errors through arithmetic operations without manual case-matching, and do-notation for division where we need to inspect the divisor before dividing.

The `binOp` helper captures the common pattern for operations that just need both sides evaluated:

```haskell
binOp :: (Int -> Int -> Int) -> Expr -> Expr -> Either String Int
binOp f a b = f <$> eval a <*> eval b
```

Division can't use `binOp` because it needs to check for zero – a decision that depends on the evaluated value, so it needs monadic style rather than applicative:

```haskell
  (Div a b) -> do
    x <- eval a
    y <- eval b
    if y == 0 then Left "Division by 0" else Right $ x `div` y
```

## The Updated REPL

The REPL now has two places where things can go wrong: parsing and evaluation. We handle them as nested cases:

```haskell
-- app/Main.hs
module Main (main) where

import Helium.Eval (eval)
import Helium.Parser (parseExpr)
import System.Console.Haskeline

main :: IO ()
main = do
  putStrLn "Helium Chapter 2 - Type an expression (Ctrl-D to quit)"
  runInputT defaultSettings repl

repl :: InputT IO ()
repl = do
  minput <- getInputLine "h> "
  case minput of
    Nothing -> outputStrLn ""
    Just input -> do
      case parseExpr input of
        Left err -> outputStrLn $ "Parse Error: " <> err
        Right expr -> do
          outputStrLn $ show expr
          case eval expr of
            Left err -> outputStrLn $ "Runtime Error: " <> err
            Right result -> outputStrLn $ show result
      repl
```

That's a lot of nesting! We could chain the parse and eval steps with do notation over the Either. But without adding transformers, that will prevent us from being able to print the AST when we need it most – when evaluation fails.

> [!NOTE]
> I played around with reimplementing this with the `mtl` library and this is what I came up with. I may revisit this approach in a later chapter.
>
> ```haskell
> liftWithPrefix :: (MonadError String m) => String -> Either String a -> m a
> liftWithPrefix prefix (Left err) = throwError (prefix <> err)
> liftWithPrefix _ (Right val) = pure val
>
> run :: (MonadError String m, MonadIO m) => String -> m Int
> run input = do
>   expr <- liftWithPrefix "Parse Error: " (parseExpr input)
>   liftIO $ print expr
>   liftWithPrefix "Runtime Error: " (eval mempty expr)
>
> repl :: InputT IO ()
> repl = do
>   minput <- getInputLine "h> "
>   case minput of
>     Nothing -> outputStrLn ""
>     Just input -> do
>       outcome <- runExceptT (run input)
>       either outputStrLn (outputStrLn . show) outcome
>
>       repl
> ```

The error labels (`"Parse Error"` and `"Runtime Error"`) tell you which phase failed. This distinction will matter more as we add more phases (like type checking).

### Trying it out

```

A parse error:

## Variables and Let Bindings

Now that the evaluator can report errors cleanly, we're ready to add variables. The expression we're aiming for is

```
let x = 5 in x + 1
```

This introduces two new concepts: **binding** a name to a value (`let x = 5`) and **referencing** a name (`x`). The evaluator will need somewhere to store these bindings – an **environment** that maps names to values.

### New AST nodes

We need two new constructors in the syntax tree:

```haskell
-- src/Helium/Syntax.hs
data Expr
  = Lit Int
  | Var String
  | Let String Expr Expr
  | Neg Expr
  | Add Expr Expr
  | Sub Expr Expr
  | Mul Expr Expr
  | Div Expr Expr
  deriving (Show, Eq)
```

`Var "x"` represents a variable reference – "look up the value of `x`." `Let "x" e1 e2` represents `let x = e1 in e2` – "evaluate `e1`, bind the result to `x`, then evaluate `e2` with that binding in scope."

Note that `Let` takes _two_ expressions: the definition and the body. This is different from how `let` works in many imperative languages, where a variable binding is a statement that persists for the rest of the block. In Helium, `let` is an expression – it has a value (the value of its body), and the binding only exists within that body. This is standard in ML-family languages.

### Parsing identifiers

Before we can parse `let` expressions, we need to parse identifiers – the names of variables. An identifier starts with a letter and is followed by zero or more alphanumeric characters:

```haskell
identifier :: Parser String
identifier = lexeme $ do
  name <- (:) <$> letterChar <*> many alphaNumChar
  if name `elem` keywords
    then empty
    else pure name
```

`(:) <$> letterChar <*> many alphaNumChar` parses one letter followed by zero or more alphanumeric characters. `lexeme` handles trailing whitespace.

The keyword check is important: without it, you'd be able to use `let` as an identifier, leading to some interesting behaviour.

```sh
h> let let = 5 in let
Parse Error: <stdin>:1:19:
  |
1 | let let = 5 in let
  |                   ^
unexpected end of input
expecting identifier

h> let let = 5 in 1 * let
Let "let" (Lit 5) (Mul (Lit 1) (Var "let"))
5
h>
```

We reject any identifier that matches a keyword with `empty` (megaparsec's silent failure). Right now the keyword list is just `["let", "in"]`. The error messages from this approach aren't great – we'll improve them later in the chapter.

We also need a parser that goes the other direction: `keyword` parses a specific word but ensures it isn't the prefix of a longer identifier:

```haskell
keyword :: String -> Parser ()
keyword name = void . lexeme $ string name <* notFollowedBy alphaNumChar
```

Without `notFollowedBy alphaNumChar`, the keyword parser for `"let"` would match the beginning of `letter`, consuming the `let` prefix and leaving `ter` behind. `notFollowedBy` checks that the next character isn't alphanumeric without consuming any input.

One more small addition: `symbol` is megaparsec's built-in helper for parsing exact strings with trailing whitespace consumption. We were using `lexeme (char ...)` and `lexeme (string ...)` in chapter 1. `symbol` is the standard way to do this:

```haskell
symbol :: String -> Parser String
symbol = L.symbol sc
```

### Parsing let expressions

With `identifier` and `keyword` in hand, `let` expressions are straightforward:

```haskell
letExpr :: Parser Expr
letExpr = do
  keyword "let"
  name <- identifier
  _ <- symbol "="
  definition <- expr
  keyword "in"
  Let name definition <$> expr
```

This reads exactly like the syntax: parse `let`, then an identifier, then `=`, then an expression (the definition), then `in`, then another expression (the body).

We also need to update `term` and `expr`:

```haskell
term :: Parser Expr
term = parens expr <|> Var <$> identifier <|> Lit <$> integer

expr :: Parser Expr
expr = letExpr <|> makeExprParser term operatorTable
```

`term` now tries to parse a variable reference (`Var <$> identifier`) in addition to parenthesised expressions and integer literals. The order between `identifier` and `integer` doesn't matter here – identifiers start with a letter and integers start with a digit, so neither parser consumes any input before failing on the other's input, and megaparsec's `<|>` backtracks freely when no input has been consumed.

`expr` tries `letExpr` first, then falls through to the operator expression parser. `let` is an expression form (not something that appears inside operator chains), so it's parsed at the `expr` level rather than as a `term`. This means `let x = 1 in x + 2` works naturally: the `in` body is a full expression that can contain operators.

### The environment

Now for the evaluator. We need a data structure to store variable bindings: an **environment**. A `Map` from names to values is the natural choice:

```haskell
-- src/Helium/Eval.hs
module Helium.Eval (eval) where

import Data.Map.Strict (Map)
import Data.Map.Strict qualified as Map
import Helium.Syntax (Expr (..))

type Env = Map String Int
```

`Map String Int` maps variable names to their integer values. We use `Data.Map.Strict` (from the `containers` package) rather than the lazy variant because we always want to evaluate values before storing them; there's no reason to defer computation here.

The evaluator now takes an environment as its first argument:

```haskell
eval :: Env -> Expr -> Either String Int
eval env expr = case expr of
  (Lit x) -> Right x
  (Var name) -> case Map.lookup name env of
    Just x -> Right x
    Nothing -> Left $ "Unbound variable: " <> name
  (Let name definition body) -> do
    x <- eval env definition
    eval (Map.insert name x env) body
  (Neg a) -> negate <$> eval env a
  (Add a b) -> binOp env (+) a b
  (Sub a b) -> binOp env (-) a b
  (Mul a b) -> binOp env (*) a b
  (Div a b) -> do
    x <- eval env a
    y <- eval env b
    if y == 0 then Left "Division by 0" else Right $ x `div` y
```

The two new cases are `Var` and `Let`:

**`Var name`** looks up the name in the environment. If it's there, return the value. If not, return an error. This is where the `Either`-based error handling from earlier in this chapter pays off – unbound variables are just another error case, reported the same way as division by zero.

**`Let name definition body`** is the interesting one. It evaluates the definition in the _current_ environment, then evaluates the body in an _extended_ environment that includes the new binding. `Map.insert name x env` creates a new map with the binding added; it doesn't modify the original. This is important: after the `let` body finishes, the binding disappears. The old environment is unchanged.

```
> let x = 5 in x + 1

eval env (Let "x" (Lit 5) (Add (Var "x") (Lit 1)))
  → eval env (Lit 5)                               = Right 5
  → eval (Map.insert "x" 5 env) (Add (Var "x") (Lit 1))
    → eval ({"x" → 5}) (Var "x")                   = Right 5
    → eval ({"x" → 5}) (Lit 1)                     = Right 1
    → Right 6
```

The `binOp` helper also needs to thread the environment through:

```haskell
binOp :: Env -> (Int -> Int -> Int) -> Expr -> Expr -> Either String Int
binOp env op a b = op <$> eval env a <*> eval env b
```

### The updated REPL

The REPL just needs to pass an empty environment to `eval`:

```haskell
-- app/Main.hs
module Main (main) where

import Helium.Eval (eval)
import Helium.Parser (parseExpr)
import System.Console.Haskeline

main :: IO ()
main = do
  putStrLn "Helium Chapter 2 - Type an expression (Ctrl-D to quit)"
  runInputT defaultSettings repl

repl :: InputT IO ()
repl = do
  minput <- getInputLine "h> "
  case minput of
    Nothing -> outputStrLn ""
    Just input -> do
      case parseExpr input of
        Left err -> outputStrLn $ "Parse Error: " <> err
        Right expr -> do
          outputStrLn $ show expr
          case eval mempty expr of
            Left err -> outputStrLn $ "Runtime Error: " <> err
            Right result -> outputStrLn $ show result
      repl
```

`mempty` is from the `Monoid` typeclass. For `Map`, it's the empty map. We could also write `Map.empty`, but `mempty` is idiomatic when you just need "the empty thing" and the type is clear from context. We can change the environment implementation and, as long as the new type satisfies the `Monoid` typeclass, we won't need to touch this code.

### Trying it out

A simple binding:

```
h> let x = 5 in x + 1
Let "x" (Lit 5) (Add (Var "x") (Lit 1))
6
```

Nested let bindings – each `let` introduces a new scope:

```
h> let x = 2 in let y = 3 in x * y
Let "x" (Lit 2) (Let "y" (Lit 3) (Mul (Var "x") (Var "y")))
6
```

An unbound variable:

```
h> x
Var "x"
Runtime Error: Unbound variable: x
```

And everything from chapter 1 still works:

```
h> 1 + 2 * 3
Add (Lit 1) (Mul (Lit 2) (Lit 3))
7
h> 1 / 0
Div (Lit 1) (Lit 0)
Runtime Error: Division by 0
```

## Improving Error Messages

The parser works, but some of its error messages are confusing. Let's look at a few cases.

### Keywords as identifiers

What happens if someone writes `let let = 5 in let`? Our keyword check in `identifier` rejects it, but since we used `empty`, the error is confusing:

```
h> let let = 5 in let
Parse Error: <stdin>:1:8:
  |
1 | let let = 5 in let
  |        ^
expecting alphanumeric character
```

`empty` is a silent failure; it tells megaparsec "this branch doesn't match" without explaining why. Megaparsec backtracks and reports the last thing it tried, which was `notFollowedBy alphaNumChar` in the keyword parser. The user has no idea the real problem is using a keyword as a variable name.

Replacing `empty` with `fail` gives megaparsec an actual error message:

```haskell
  if name `elem` keywords
    then fail $ "keyword " <> show name <> " cannot be used as an identifier"
    else pure name
```

```
h> let let = 5 in let
Parse Error: <stdin>:1:8:
  |
1 | let let = 5 in let
  |        ^
keyword "let" cannot be used as an identifier
```

Better – the message is clear. But the caret points past the end of the keyword. By the time `fail` fires, megaparsec has already consumed `let` and the position has advanced. We can fix this by capturing the offset before parsing the name and rewinding it before the error:

```haskell
identifier :: Parser String
identifier = lexeme $ do
  pos <- getOffset
  name <- (:) <$> letterChar <*> many alphaNumChar
  if name `elem` keywords
    then do
      setOffset pos
      fail $ "keyword " <> show name <> " cannot be used as an identifier"
    else pure name
```

```
h> let let = 5 in let
Parse Error: <stdin>:1:5:
  |
1 | let let = 5 in let
  |     ^
keyword "let" cannot be used as an identifier
```

Now the caret points to column 5: the start of the offending keyword.

### Missing identifier after let

What about `let = 5 in x`?

```
h> let = 5 in x
Parse Error: <stdin>:1:5:
  |
1 | let = 5 in x
  |     ^
unexpected '='
expecting letter
```

"Expecting letter" is technically accurate – identifiers start with a letter – but it's not helpful. A user seeing this has to work backwards from "letter" to "oh, I forgot the variable name."

Megaparsec's `<?>` operator replaces the "expecting" label on a parser. We can add it in `letExpr`:

```haskell
  name <- identifier <?> "identifier"
```

```
h> let = 5 in x
Parse Error: <stdin>:1:5:
  |
1 | let = 5 in x
  |     ^
unexpected '='
expecting identifier
```

A small change, but the error message now speaks in terms the user understands rather than in terms of the parser's implementation.

### When the defaults are good enough

Not every error needs custom handling. Consider a typo in the `in` keyword:

```
h> let x = 5 at x
Parse Error: <stdin>:1:11:
  |
1 | let x = 5 at x
  |           ^^
unexpected "at"
expecting "in", '*', '+', '-', or '/'
```

This is excellent without any extra work. After parsing the definition `5`, megaparsec expects either the `in` keyword or an operator (since `5` could be the start of a larger expression like `5 + 1`). It found `at` instead, and lists all the valid alternatives. No label needed.

The lesson: add custom error handling where megaparsec's defaults are genuinely confusing, and leave them alone where they're already clear. Over-labelling can make error messages _worse_ by hiding useful information about what alternatives were expected.

## What we built

This chapter added two things: error handling and variables. The evaluation error handling (switching `eval` from `Expr -> Int` to `Expr -> Either String Int`) gave us a uniform way to report both division by zero and unbound variables.

The variable system introduced the **environment**, a map from names to values that gets threaded through evaluation. `let` extends the environment for its body; `Var` looks up names in it. The key point is that `Map.insert` doesn't mutate – it returns a new map, so each `let` body sees its own bindings without affecting the outer scope.

| --------------- | ------------------------------------------------------------------- |
| `Helium.Syntax` | Added `Var` and `Let` constructors                                  |
| `Helium.Parser` | Added `identifier`, `keyword`, `letExpr`; updated `term` and `expr` |
| `Helium.Eval`   | Added `Env` type; `eval` takes and threads an environment           |
| `Main`          | Passes `mempty` as initial environment                              |

In the next chapter, we'll add functions: lambdas and application. That's where the evaluator starts getting interesting: we'll need **closures** to capture the environment at the point where a function is defined, not where it's called.
